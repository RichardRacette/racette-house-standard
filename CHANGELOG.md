# Changelog

All notable changes to the Racette House Standard (RHS) are documented in this file.

Unlike a traditional software changelog, this document records significant milestones in the evolution of the methodology, governance, and research rather than every individual edit.

Minor corrections, formatting improvements, and editorial revisions are preserved through Git history.

---

# Purpose

The purpose of this changelog is to preserve the historical evolution of the Racette House Standard.

Every published edition should explain not only **what changed**, but **why the methodology changed**.

This document is intended to become the readable historical record of RHS as it matures over time.

---

# Publication Rhythm

RHS follows an edition-based publication rhythm rather than a software release cycle.

Each edition represents a meaningful advancement in the development of the methodology.

Version numbers identify editions and communicate the maturity of the standard—not the amount of work performed.

The expected progression of the early editions is:

| Version | Edition Focus |
| --- | --- |
| **v0.1.0** | Repository Foundation |
| **v0.2.0** | Governance Framework |
| **v0.3.0** | Documentation Standards |
| **v0.4.0** | Household Knowledge Language (HKL) Alpha |
| **v0.5.0** | Reference Record Framework |
| **v0.6.0** | Validation Framework |
| **v0.7.0** | Research Methodology Refinement |
| **v0.8.0** | Archive Expansion |
| **v0.9.0** | Publication Candidate |
| **v1.0.0** | First Stable Edition |

This sequence is intentionally flexible.

Research findings may introduce additional editions, combine planned editions, or alter the order of future work.

---

# Versioning Philosophy

RHS follows semantic versioning in spirit rather than strict software practice.

Version numbers identify published editions and communicate the evolution of the methodology.

## Major Versions

Major version changes are reserved for stable editions that substantially advance or restructure the methodology or governance of RHS.

Examples include:

- the first stable edition
- major methodological revisions
- substantial governance restructuring
- changes that alter how the standard should be applied

## Minor Versions

Minor version changes represent meaningful editions that introduce new standards, capabilities, frameworks, or validated concepts.

Examples include:

- new governance documents
- HKL milestones
- Reference Record standards
- validation frameworks
- publication workflows
- major archive-based research findings

## Patch Versions

Patch versions are reserved for clarifications, corrections, editorial improvements, and other non-substantive revisions that do not materially alter the methodology.

Patch editions should improve clarity while preserving historical continuity.

# First Edition

## Version 0.1.0 — Repository Foundation

**Status:** Active Research

### Added

#### Repository Foundation

- Established the public GitHub repository for the Racette House Standard.
- Defined the initial repository structure.
- Adopted Git as the permanent revision history of the project.

#### Foundational Documents

- Published `README.md`.
- Published `ROADMAP.md`.
- Published `PHILOSOPHY.md`.
- Published `CHANGELOG.md`.

#### Research Identity

- Defined RHS as a research standard rather than a software project.
- Identified the Racette House Archive as the reference implementation.
- Established the guiding objective of preserving human understanding rather than information alone.
- Established stewardship, responsible development, and preserved attribution as core project commitments.

#### Project Architecture

- Distinguished the Racette House Standard from the Household Knowledge Language and the Racette House Archive.
- Established the initial three-layer architecture:
  - RHS as the methodology
  - HKL as the representation language
  - the Racette House Archive as the reference implementation

#### Governance

- Began establishing the governance framework that will guide future standards development.
- Retained project stewardship under its original author during the active research stage.
- Deferred adoption of an open-source license until the project’s long-term governance and publication model are more mature.

# Criteria for a New Edition

A new edition should normally be published when one or more of the following occur:

- A foundational standard is completed.
- Governance materially changes.
- HKL gains significant new capabilities.
- Validation research produces methodological improvements.
- The archive demonstrates new classes of preserved understanding.
- A major research phase reaches a meaningful milestone.
- A stable body of RHS documentation becomes appropriate for long-term reference.

Routine edits should remain part of Git history rather than generating a new edition.

Patch editions may be used when a group of corrections or clarifications should be formally preserved and identified.

---

# Edition Records

Each edition entry should include, when applicable:

- edition name
- version number
- publication date
- status
- documents added
- documents revised
- methodological changes
- governance changes
- research findings
- known limitations
- reasons for publication

The changelog should summarize the significance of an edition rather than reproduce its complete commit history.

---

# Historical Integrity

Published editions are historical artifacts.

Later editions should improve the methodology without rewriting the historical record.

Corrections should be documented transparently.

Superseded ideas should remain discoverable when they are important to understanding how RHS evolved.

The evolution of RHS should remain understandable to future readers.

**The history of the methodology is itself part of the methodology.**
